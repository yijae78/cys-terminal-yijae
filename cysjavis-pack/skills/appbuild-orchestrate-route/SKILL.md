# appbuild-orchestrate-route

검증 리포트의 문제를 **올바른 모듈로 보낸다.** 루프의 핵심: 메인은 절대 직접 수정하지 않고
판별·라우팅만 한다 — 그래서 메인은 지능이 높아야 한다(프론트/백 판별 책임).

## 절차

1. **원인 판별** → 검증: 각 결함이 프론트 문제인지 백엔드 문제인지(때로 둘 다·계약 문제)를
   증상·재현으로 판별. 잘못 라우팅하면 수정이 헛돈다.
2. **수정 위임** → 검증: 프론트 결함 → 프론트 페인, 백 결함 → 백 페인에 수정 위임(작업·증상·
   기대 결과 전달). 병행 가능한 건 동시에.
3. **재검증 트리거** → 검증: 수정 완료 리포트를 받으면 `[[appbuild-orchestrate-verify]]`로
   재검증을 시킨다(메인이 직접 확인 아님 — 다른 페인 증거).
4. **루프 상태** → 검증: 라우팅·수정·재검증 회차를 `loop-state.json`에 기록.

## 출력 계약

라우팅 결정(결함→모듈) + 위임 결과 + 재검증 트리거. 상위 `[[appbuild-orchestrate]]`로 반환.
메인의 직접 수정·오라우팅(프론트 문제를 백에)은 루프를 망친다 — 판별 정확성이 생명.

**증거 규약 (웹/앱 UI 매체)**: 입력으로 받는 검증 findings와 수정 위임 브리프는 웹/앱 UI 증거
3종(스크린샷 경로 · DOM ref · 코드 추정 위치[**미상이면 '미상' 명시**])을 담아야 한다 — 원인
판별·라우팅은 이 증거의 DOM ref·코드 추정 위치를 근거로 삼는다. 증거 없는 결함은 라우팅 전
`[[appbuild-orchestrate-verify]]`로 반려. 위치 ref 규약 `경로.png#ref_N`·상세 =
`${CYS_PACK_DIR:-$HOME/.cys/pack}/round/EVIDENCE_CONVENTION.md`(§1 웹/앱 UI).
